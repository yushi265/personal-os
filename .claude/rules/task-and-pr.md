# タスク分割と PR 運用

## 実装着手前: タスク分割

実装指示を受けたら、コードを書き始める**前に**:

1. 変更を**ユニット（Unit of Work）**として捉える（データ層→表示層まで縦に貫く 1 機能 ＝ 1 チケット。単一レイヤーのプロジェクトもあり得る。用語は [glossary.md](../../docs/ai-dlc/glossary.md)）。実装の段取りは順序付きタスク／ボルトに分けてよいが、**PR の単位はハーネスで固定しない**。コミット／PR の対象は **Gate 3（コミット承認）で人間が決める**。

### PR の目安（規約ではない）

- **PR の粒度はハーネスで固定しない**（行数上限も、ユニット／ボルト単位の強制もしない）。最終判断は Gate 3。
- 目安: サイズは行数ではなく**レビュー可能性**で判断する。
- 各 PR は意味のあるまとまり（単一責務）に絞る。
- 動作に必要な配線（設定/型/定数）は同一 PR に含める。
- レビュー不能な中途半端な分割（レイヤー途中での切断など）はしない。

## 完了の定義（DoD）と品質ゲート

PR 作成前に、**必ず**以下を実行し pass を確認する。失敗したら直してから進む。

具体コマンドはプロジェクトが定義する。ハーネスが束ねる pre-commit ゲート（`lefthook`）と secret 検知（`gitleaks`）はスタック非依存の共通機構として使う。

| 対象 | コマンド（例） |
|------|---------|
| pre-commit 一括 | `npx lefthook run pre-commit`（secret 検知 = `gitleaks` + 各レイヤーの lint/フォーマット/テストを束ねる。フックの中身はプロジェクトが定義） |
| 各レイヤー | フォーマッタ（差分なし）/ linter / 型チェック / テスト（プロジェクトが定めるコマンド） |

- テスト追加状況・カバレッジ方針は [testing.md](./testing.md) に従う。
- spec があるタスクは AC を満たしていることを確認する（[spec-driven.md](./spec-driven.md)）。
- PR では CI（GitHub Actions 等）が同等チェック（各レイヤーの lint/テスト + secret 検知）を再実行する。ローカル hook を経由しないコミットの素通りを防ぐ。

## コミット / ブランチ / PR

- ブランチ: チケットありは `feature/<TICKET>-簡潔な説明`、なしは `feature/簡潔な説明`。保護ブランチへの直接 PR（ベースブランチ運用）は CI（ブランチ運用チェック）でブロックする（設定はプロジェクト依存）。
- コミット（Conventional Commits）:
  - チケットあり: `<type>(<TICKET>): 変更内容`（`feat` / `fix` / `refactor` / `test` / `docs` / `chore` / `perf` / `ci` / `style`）
  - チケットなし: `<type>: 変更内容`
- `-f`（強制）系コマンドは使用しない。hook をスキップ（`--no-verify` 等）しない。
- push: `git push -u origin HEAD`。PR は `gh pr create`（テンプレートがあれば従う）。

## レビュー

- レビューは人間 + AI レビュアーが担う（AI アダプタ文書があればその規約に従う）。
- レビュー指摘は重大度順に対応し、品質ゲート再実行 → 指摘ゼロまで繰り返す。
- 設計影響のない bugfix / 外部仕様不変のリファクタ / テスト追加のみの場合、spec の更新は省略可。省略時は PR 本文に `spec更新なし: <理由>` を 1 行記載する。
