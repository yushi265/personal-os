# テスト方針（TDD / カバレッジ）

## 原則

- **テスト駆動開発（TDD）を基本とする**。RED → GREEN → REFACTOR の順で進める。
- 実装に対応するテストを**同一 PR**で追加する。
- 目安カバレッジは **80%+**。
- **挙動変更を伴わない変更**（typo / コメント / ドキュメントのみ / フォーマットのみ。Tier 3 の該当ケース）では TDD 不要・テスト追加不要。Stage 宣言と self-review で「コード挙動変更なし」を N/A 根拠として記録する（[risk-tiers.md](./risk-tiers.md)）。

## レイヤー別ツール

テストランナー・カバレッジ計測ツールは各レイヤー/サブツリーがローカル規約で定める（例: 言語標準のテストコマンド、テーブル駆動（パラメータ化）を基本とする網羅、同ツールのカバレッジ計測）。

| 対象 | テスト | カバレッジ |
|------|--------|-----------|
| 各レイヤー | プロジェクト/レイヤーが定めるテストランナー（テーブル駆動を基本） | 同ランナーのカバレッジ計測 |

> レイヤー/サブツリーごとにローカル規約があればそれを優先する。ここでは横断方針を扱う。

## テスト観点（最低限）

- テストケースは技法マッピング表（下記）で導出する（「正常系・異常系・境界値」の具体化）。
- 失敗系（外部 API エラー、認証失効、入力不正）を実装と同時に扱う。
- アクセス境界（他ユーザー / 他組織のデータが見えない等。該当する場合）を要する機能は、その分離をテストで担保する。

## 技法マッピング表（ケース導出の正本）

どの技法を使うかは**対象の形で機械的に決まる**。
テストリスト作成ステップ（[tdd-cycle](../skills/tdd-cycle/SKILL.md)）はこの表に照らしてケースを導出する。

| 対象の形 | 適用技法 | 導出するケース |
|---------|---------|--------------|
| 値域・長さ・フォーマット制約のある入力 | 同値分割 + 境界値分析 | 有効/無効クラス各 1 + 各境界とその前後 |
| 複数条件の組み合わせで結果が変わる判定 | デシジョンテーブル | ルール表の全列（矛盾・抜けの検出込み） |
| 状態・ライフサイクルを持つ対象 | 状態遷移テスト | 全有効遷移 + 代表的な禁止遷移 |
| 独立パラメータが多く組み合わせ爆発する | ペアワイズ | 2 因子網羅に削減 |
| 上記に該当しない単純な振る舞い | 代表値 1 ケース | 正常系 1 + 失敗系（あれば） |

## テスト種別の分担（単体 / レイヤー内結合）

どのテスト種別で何を担保するかの基準。spec のテスト戦略（`index.md`）はこの基準に従い、
機能ごとの担保場所はテストマトリクス（AC × 種別 × レイヤー）で示す。

> 種別ごとのツールはプロジェクトが選定する（例: レイヤー内結合 = API テスト / 実依存を立てた結合 / UI コンポーネント結合）。

| 種別 | 担保するもの | 担保しないもの |
|------|------------|--------------|
| レイヤー内結合 | AC の正常系フロー全件・代表的な認証ガード・異常系・境界値（認証 / 権限 / 未検出 / サーバエラー / 期限切れ等）・エラー変換の全分岐・アクセス境界の分離（データ層レベル） | — |
| 単体 | ロジック・変換（mapper 等）・コンポーネント表示 | — |

- 正常系の縦フローはレイヤー内結合テストで担保する（重複分のロジック・変換検証は単体に置く）。
- 時間経過・外部障害・改ざん入力・データ層直の分離保証は、レイヤー内結合または単体で必ず担保する。
- アプリ全体を起動する E2E は AI-DLC のデフォルト Stage には含めない（Stage 7 は廃止）。必要なプロジェクトは任意で追加してよいが、本数を絞り flaky を放置しないこと（正常系の担保はレイヤー内結合が正・E2E で重複させない）。

## 禁止

- テストを「後で書く前提」で本実装をマージすること。
- 実装に合わせてテストを甘くすること。テストではなく実装を直す（テスト自体が誤りの場合を除く）。
