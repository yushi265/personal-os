# 仕様書駆動（spec-first）

このルールは、実装着手前に spec を読むことを強制する。

## 実装着手前に必ず行う

0. まずリスクティア（[risk-tiers.md](./risk-tiers.md)）を判定する（spec の要否と人間ゲートの深さが決まる）。
1. タスクにチケット番号が紐づくか確認する。紐づく場合は `import-ticket` スキルでチケット本文・AC を取得し、**要件の源泉**とする（チケット → spec → 実装の連鎖）。
2. 紐づく場合: `docs/spec/<TICKET>-*/` ディレクトリを探索し、`index.md`（入口）から関与レイヤーのファイル（プロジェクトが定義するレイヤー別 `.md`。例: `data.md` / `service.md` / `ui.md`）を読む。これが実装の正本（AC・対象範囲・制約）。
3. spec が **無い / 曖昧 / AC が観測不能** な場合は、**コードを書き始めず**に次のいずれかを行う:
   - 人間に確認する、または
   - `docs/spec/_TEMPLATE/` を雛形に spec を作成し（`create-spec` スキル）、人間の合意を得る（Gate 2 委任時は要点提示のうえ Gate 3 で事後確認: [risk-tiers.md](./risk-tiers.md) の「Gate 2 委任」）。
4. spec に要約された制約・AC・対象範囲を理解する。
5. そのチケットの spec だけでなく、**関連する既存実装・既存機能の spec・共有契約（データスキーマ / API 契約 / 共有コンポーネント）も把握する**。新機能が既存資産と重複・衝突・依存しないかを着手前に確認する（現実装の調査手順は `create-spec` スキルが担う）。対象は旧システムではなく **このプロジェクト内の現実装**。

## spec 省略の条件（単一適応フローでの🔓条件付きスキップ）

spec の要否はリスクティア（[risk-tiers.md](./risk-tiers.md)）で決まる: **Tier 1 = 必須 / Tier 2 = 下記 4 条件で判定 / Tier 3 = 不要**。
本節（4 条件）は Tier 2 の「条件付き」判定の本文（正本）であり、risk-tiers.md はこれを参照する。

AI-DLC は単一適応フロー（[ai-dlc-flow](../skills/ai-dlc-flow/SKILL.md)）で進める。spec 作成（Stage 2）は🔓条件付き Stage であり、
**次を全て満たす時だけ**省略してよい。1つでも外れたら spec 必須。

1. 単一レイヤーで完結する（プロジェクトが定義するレイヤーのいずれか 1 つ。例: data / service / ui）。
2. レイヤー間インターフェース（データスキーマ / API 契約（JSON 等）/ ルーティング等）を変更しない。
3. データスキーマ・API 仕様・認証/認可・ロール体系・データ境界に触れない（＝「変更コストが高い判断」を含まない）。
4. 低リスク（容易にロールバックでき、影響範囲が局所）。

> Tier 2 では条件②③はティア定義（Tier 1 要素に触れない）から自動的に充足する。実質の判定は**①（単一レイヤーか）と④（低リスクか）**。複数レイヤーの変更は①を満たさないため spec 必須になる。

- **Tier 2 で省略する場合**は、Stage 宣言（Gate 1）の理由欄に判定ティア（[risk-tiers.md](./risk-tiers.md)）と上記4条件の充足を明記する。**Tier 3** はティア判定根拠の明記のみでよい（4 条件判定は不要）。
- 省略しても TDD（[testing.md](./testing.md)）の必須 Stage は外れない。
- spec を省略してよいか判断に迷う場合は、省略せず spec を作る（安全側に倒す）。

## spec の位置づけ

- 設計方針・規約（Why/What）はハーネス（`docs/architecture.md`・横断 `.claude/rules/`・各レイヤーの docs（プロジェクトで用意））が持つ。変更は慎重に、必要なら ADR を伴う。
- 実装 spec（How/What exactly）= `docs/spec/<TICKET>-*/`（チケット単位ディレクトリ・レイヤー別ファイル）。実装に合わせて更新する。
- spec はハーネスの設計方針・規約に**従う**。spec で方針を上書きしない。矛盾を見つけたら人間へ報告する。

## spec の更新

- `docs/spec/` の実装 spec は、実装 PR と**同一 PR**で作成・更新する。
- 設計方針そのものの変更が必要な場合は、実装を止めて人間と合意してから進める。

## 禁止

- spec を読まずに実装を始めること。
- 仕様の不足・曖昧さを推測で埋めて実装を進めること（必ず人間へ確認）。
