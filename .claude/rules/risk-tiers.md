# リスクティア（人間ゲートの深さを決める）

タスク着手時（Stage 0+1 の Stage 宣言）に必ずティアを判定し、Stage 宣言に明記する。
判定に迷ったら**上のティアに倒す**（安全側）。

ティアは AI-DLC 単一適応フロー（[ai-dlc-flow](../skills/ai-dlc-flow/SKILL.md)）の人間ゲート（Gate 1/Gate 2/Gate 3）の深さを決める。
ティアが変えるのは**人間ゲートのブロッキング有無と spec 要否のみ**。Stage の条件付きスキップ基準（spec 等）は
ai-dlc-flow の「必須 Stage と条件付きスキップ基準」に従う（ティアで変わらない）。

## ティア定義

| ティア | 対象（いずれか 1 つでも該当で昇格） | 人間ゲート | spec |
|--------|-----------------------------------|-----------|------|
| **Tier 1（高リスク・事前承認必須）** | 変更コストが高い領域（例: DB/データスキーマ / 認証・認可 / データ境界 / レイヤー間インターフェース（API 契約・スキーマ・ルーティング等）/ ロール体系 / 外部公開 API）。プロジェクトの「変更コストが高い判断」に合わせて具体化する | Gate 1 + Gate 2 + Gate 3 すべてブロッキング承認。spec 省略不可（Gate 2 は委任可: 下記運用節） | 必須（ADR が要る場合あり） |
| **Tier 2（中リスク・要点承認）** | **Tier 1 に該当しない機能追加・変更（レイヤー数問わず）** | Gate 1 + Gate 3 はブロッキング承認。Gate 2 は spec 実行時のみ（委任可: 下記運用節） | 条件付き（[spec-driven.md](./spec-driven.md) の 4 条件で判定） |
| **Tier 3（低リスク・自走 + 事後承認）** | typo / コメント / ドキュメントのみ（**ハーネス文書 = `.claude/` 配下・各 `CLAUDE.md`・`AGENTS.md` 等の AI 向けアダプタ文書を除く。ハーネス文書の変更は Tier 2 以上**）/ ログ追加 / テスト追加のみ / 設定値の局所変更（ロールバック容易）。**いずれも Tier 1 要素に触れないこと** | Gate 1 は**宣言のみ（承認待ち不要）**。Gate 3（コミット前承認）のみブロッキング | 不要 |

- Tier 2 の「条件付き」は、[spec-driven.md](./spec-driven.md) の「spec 省略の条件」（🔓条件付きスキップの 4 条件）と同一。本文はそちらが正本（ここでは二重定義しない）。

## 運用

- 判定は **Tier 1 → Tier 3 → 残りは Tier 2（catch-all）** の順で行う。
- ティア判定の根拠（どの行に該当するか）を Stage 宣言に 1 行で書く。
- Tier 1 で条件付き Stage（spec 等）をスキップする場合も、その判断理由を Gate 1 承認に含める。
- Tier 3 の Stage 宣言・判定根拠は progress.md ではなく会話で宣言し、Gate 3 で再提示して事後確認を受ける（Tier 3 は progress.md を作らない）。
- 作業中に Tier 1 要素に触れることが判明したら、**即座に停止して再宣言**する（ティアの昇格はいつでも、降格は人間承認時のみ）。
- **Tier 3 の TDD 適用**: コード挙動変更を伴う変更（バグ修正・ロジック追加・テスト追加・設定値の挙動影響あり）では TDD 必須。**挙動変更を伴わない変更**（typo / コメント / ドキュメントのみ / フォーマットのみ）では TDD 不要・テスト追加不要。Stage 宣言で「コード挙動変更なし」を理由として記録する（self-review でも N/A 根拠として再提示）。
- 必須規約（TDD 等）はティアに関わらず免除されない（Tier 3 でも破れない）。プロジェクトがセキュリティ等の必須規約を `.claude/rules/` に追加した場合も同様。

## Gate 2 委任（オプトイン・定義の正本は [glossary.md](../../docs/ai-dlc/glossary.md)）

Tier 1/2 の spec 実行ボルトで、**Gate 1 で人間が明示的にオプトインした場合に限り**、Gate 2 を承認待ちなし（declare-only 相当）へ降格できる。

- **発動条件**: Gate 1 承認時の明示選択のみ。依頼文での事前指定も可とするが、Tier 1/2 の Gate 1 はブロッキングなので必ず人間の目を通ってから発効する（無人発効の経路は無い）。
- **デフォルトは非委任**。AI から委任を勧めない（Gate 1 で選択肢として中立に提示するのみ）。
- 委任時も spec は**必ず作成**し、要点サマリ（契約の具体値 + Why + UI/UX 方針 + テストケース一覧）を**必ず提示**する。変わるのは**承認待ちの有無だけ**。提示後は承認を待たず Stage 3+4 へ進み、**Gate 3 で実装とまとめて事後確認**を受ける（委任後に doc を改訂した場合はその差分も再提示）。
- **Gate 3 は全ティアで委任不可**（コミット承認は死守）。
- **ティア再宣言（昇格）時は委任もリセット**し、再オプトインを求める（engine は `gate-undelegate gate2` で追随）。
- **差し戻しコストの受容が trade-off**: Gate 3 で spec に NG が出たら spec 修正 → 実装差し戻しになる。差し戻しが痛いボルトでは委任しない。Tier 1 の委任は「Stage 0+1 で変更コストの高い判断が確定済み」が前提（運用を Tier 2 から始めて実績を見て Tier 1 へ広げる段階導入も可）。
- 委任は progress.md ヘッダの「Gate 2 委任」行と engine audit（`report <state> gate-delegate gate2`・advisory）に記録する。
