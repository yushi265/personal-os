# 実装全体監査（節目レポート）手順

実装**全体**を spec・ハーネスと突き合わせ、全体の適合状態を点検したレポートを作る。

| | `self-review` | `impl-audit`（本スキル） |
|---|---|---|
| スコープ | 今回の変更（diff） | 実装全体（spec の対象範囲） |
| タイミング | 毎ボルトのコミット提案前 | 節目（大きな区切り） |
| ゴール | Must ゼロまで**直す** | 点検レポートを**作る**（直さない） |

## 0. 入力

- **チケット ID（必須）**: 例 `PROJ-001`。`docs/spec/<TICKET>-*/` を解決する。1 評価 = 1 spec。
- spec が見つからなければ中断して人間に確認する。

## 1. 実行: impl-auditor へ委譲

評価は **`impl-auditor` サブエージェント（単一・読み取り専用）** に委譲する。
レイヤー別に分割しない（レイヤー間契約の整合を 1 つの目で見ることが全体監査の価値）。
エージェントへの指示にはチケット ID と本スキルのパス（レポート様式の正本）を含める。

## 2. 評価軸（4 軸）

1. **要件適合**: spec の AC を 1 件ずつ実装・テスト（レイヤー内結合 / 単体）に対応付ける（AC ↔ 証跡の対応表。充足 n/m）
2. **設計適合**: **全体（横断）とレイヤーごとの 2 段構え**で監査する。指摘は Must / Should / IMO で分類
   - **2a. 全体（横断）**: レイヤー間の依存方向・レイヤー間契約の一気通貫整合（スキーマ ↔ API ↔ 画面）・
     横断規約（`.claude/rules/`: simplicity / testing）
   - **2b. レイヤーごと**: spec の対象レイヤーそれぞれを、当該レイヤーの規約 docs（プロジェクトで用意）と突き合わせて個別判定する（レイヤー名・観点はプロジェクトに合わせる。以下は例）
     - データ層: スキーマ分離・データ境界（アクセス制御）・マイグレーション規約
     - サービス層: モジュール境界・レイヤーパターン・エラー/ログ設計
     - 表示層: ルーティング/認証・コンポーネント設計・状態管理・コーディング規約
3. **品質実測**: テスト・カバレッジ・静的解析を**評価時に実行**した実測値（過去の報告を信用しない）
4. **残課題・負債の棚卸し**: 実装上のモック・省略・スコープ外の一覧と、要対応事項
   （spec の「やらないこと」・コード内の「TODO」「あとで」「モック」コメント・未導入ツールを収集し、優先度: 高/中/低を付ける）

## 3. 実測の規則

- ユニット/結合・静的解析（プロジェクトのテスト / lint / 型チェック）は**実行して**証跡を取る。
- E2E は **任意**（AI-DLC のデフォルト Stage には含めない・[testing.md](../../rules/testing.md)）。プロジェクトが E2E を持ち、かつ
  **E2E 環境が起動済み（全サービス healthy）の場合のみ実行**する。未起動なら **自分で起動せず**、
  「未実測（環境未起動）」とレポートに明記する。E2E を持たないプロジェクトは N/A。
- 評価中にコードを**直さない**（evaluator は Write/Edit を持たない）。発見した問題は指摘として記録する。

## 4. 総合判定（機械的に適用する）

| 判定 | 条件 |
|---|---|
| **健全 ✅** | Must 指摘ゼロ かつ AC 全充足 かつ 高優先の残課題なし |
| **要対応 ⚠️** | Must 指摘ゼロ かつ AC 全充足 だが、高優先の残課題あり（要対応リスト付き） |
| **重大 ❌** | Must 指摘あり または AC 未充足あり |

## 5. レポート様式

```markdown
# <TICKET> 実装監査レポート（<YYYY-MM-DD>）

# 総合判定: <健全 ✅ | 要対応 ⚠️ | 重大 ❌>
<判定根拠を 2〜3 行>

## 1. 要件適合（AC 充足 n/m）
| AC | 内容（要約） | 証跡（テスト名〔レイヤー内結合 / 単体〕・実装箇所。E2E シナリオがあれば併記） | 判定 |

## 2. 設計適合
### 2a. 全体（横断）
<判定 ✅/指摘 + 依存方向・レイヤー間契約の整合・横断規約の根拠>
### 2b. レイヤー別
<spec の対象レイヤーごとに小見出し（#### データ層 / #### サービス層 / #### 表示層 等、プロジェクトのレイヤー名）で記載する。**表は使わない**>
#### <レイヤー名> — <判定 ✅/指摘あり>
- 突き合わせた規約: <当該レイヤーの規約 docs>
- <根拠・指摘の箇条書き（Must / Should / IMO、ファイル:行つき）>

## 3. 品質実測（実行日時: <timestamp>）
| 対象 | コマンド | 結果 |
<E2E は任意。実施した場合のみ結果を記載し、未実施 / 環境未起動ならその旨と理由を明記>

## 4. 残課題・負債の棚卸し
| # | 要因 | 優先度 | 要対応内容 |

## 申し送り（IMO・補足）
```

## 6. 保存

- evaluator の最終出力をメインループが受け取り、要約を人間に提示する。
- 人間の確認後、レポート全文をプロジェクトの**ローカル feedback 保存先（例: `z-feedbacks/<YYYYMMDD>-<ticket>-eval.md`）** に保存する。
  この保存先は**ローカル運用**（gitignore しておき、コミットしない）。
  リポジトリの公式記録に残す必要が出た場合は人間の指示で別途扱う。

## 禁止

- 評価中の実装修正（評価と修正を混ぜない。修正は評価後に別タスクとして行う）
- 環境の作り替え（E2E 環境の起動 / 破棄。E2E は任意で、評価者は自ら起動しない）
- 判定基準（4 章）を恣意的に曲げること。基準自体を変えたい場合は本スキルを人間合意の上で更新する
