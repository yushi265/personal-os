# チケット取り込み（チケット → 要件入力）手順

[ai-dlc-flow](../ai-dlc-flow/SKILL.md) Stage 0+1 の最初の一手。**チケットを要件の源泉**として取り込み、要件整理の入力に整形する（チケット → spec → 実装の連鎖の起点）。

## 0. 入力

- チケットの **URL またはキー**を引数で受け取る（例: Jira なら `https://<org>.atlassian.net/browse/PROJ-123` / `PROJ-123`）。
- URL/キーから **チケットキー**（`PROJ-123` 等）と **プロジェクトキー**（`PROJ`）を抽出する。
  これが spec ディレクトリ名 `docs/spec/<TICKET>-<kebab-case-feature>/` とブランチ名の背骨になる。
- 引数が無ければ人間にチケット URL/キーを尋ねる（チケットレスで進める可否は ai-dlc-flow のティア判定に従う）。

## 1. チケット取得（チケット管理システム）

> 以下は Jira/Atlassian MCP を使う場合の例。別のチケット管理システムなら同等の取得手段に読み替える。
> チケット管理システムを使っていなければ本スキルは N/A（人間からチケット相当の要件を直接受け取る）。

1. Atlassian MCP の取得系ツールが使えるか確認する。**未認証なら** `mcp__claude_ai_Atlassian__authenticate` で
   認証 URL を取得 → 人間に共有 → ブラウザ認可後のコールバック URL を `mcp__claude_ai_Atlassian__complete_authentication` に渡す。
2. `ToolSearch`（query 例: `jira get issue atlassian`）で取得系ツール（`getJiraIssue` / `getAccessibleAtlassianResources` 等。
   名称は環境で変わるため都度解決する）を解決し、チケットを取得する。
3. 取得する項目: 要約（summary）・説明（description）・**受け入れ基準（AC。専用フィールド or 本文中）**・
   課題タイプ・ラベル/コンポーネント・関連課題（リンク）・添付・コメントの要点。
4. MCP が使えない/権限が無い場合は**中断せず**、人間にチケット本文の貼り付けを依頼してフォールバックする。

## 2. 要件入力への整形

取得内容を、Stage 1 が消費できる構造へ正規化する（**推測で埋めない**。原文に無い AC を捏造しない）。

- **概要**: 1〜3 行。
- **背景 / 目的**: なぜやるか。
- **AC 候補**: チケットの AC を観測可能な粒度の候補として列挙（確定は Stage 1 / spec）。
- **対象レイヤー推定**: プロジェクトのどのレイヤーに及びそうか（確定は spec）。
- **スコープ外候補**: チケットに「やらない」記述があれば拾う（YAGNI）。
- **不明点 / 未決事項**: チケットだけでは確定できない点を列挙 → Stage 1 の AskUserQuestion 材料にする。
- **関連**: リンク課題・依存・既存機能への影響の手掛かり。

## 3. 受け渡し

- 整形結果を要約して提示し、ai-dlc-flow の Stage 0+1（要件整理＋Stage 宣言）へ引き渡す。
- チケットキーを `docs/spec/<TICKET>-*/` の命名・ブランチ名に使う（spec を実行する場合）。
- **複数ユニットに割れる**と判断したら、その旨を Stage 0 に申し送る（ユニット計画の記録は spec の `index.md` ← create-spec）。

## 禁止

- チケット本文に無い AC・制約を推測で補完して確定すること（不明点は人間へ確認）。
- 取得した sessionId / トークン等の秘匿情報をログ・spec・コミットに書くこと。
- チケット取得の失敗を握りつぶし、要件が曖昧なまま実装に進むこと。
