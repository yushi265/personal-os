# retro note 棚卸し手順（Try → ハーネス改善）

retro note（`docs/ai-dlc/retro/<TICKET>.md`）にたまった未対応 Try を集約し、
ハーネス改善タスクとして人間に提案 → 承認結果を各 note に反映する。
トリガー条件・剪定チェックの内容の正本は [docs/ai-dlc/retro/README.md](../../../docs/ai-dlc/retro/README.md)。

## 重要原則

- **このスキルはユーザーの明示的な指示で実行する。AI が自動で棚卸しを開始しない**
  （トリガー達成を検知した場合も、実施を促すに留める）。
- **ステータス更新・記録追記より先に必ず人間の承認を得る**（手順③の承認が先・書き込みは後）。

## 手順

### 1. 未対応 Try の集約

- `docs/ai-dlc/retro/` の全 retro note（`_TEMPLATE.md`・`README.md` を除く）を読む。
- 各 note の「フロー改善アクション」表から **ステータス=未対応** の行を集約する
  （Try / 還流先 / 出典 note の一覧表にまとめる）。

### 1.5 前回 Try の効果測定（再発チェック）

正本は [retro/README.md](../../../docs/ai-dlc/retro/README.md) の「効果測定」。`pnpm -C .claude/aidlc learnings measure` を実行し、**再発マトリクス**（カテゴリ × 採用 Try 期間の Problem 出現数 + 出典 note + sensor FAIL 推移）を材料にする（note 全読みの手動照合は「measure 出力の検証」に降格）。判定は人間が下す: 採用 Try の日付より後のバケットに同カテゴリの Problem があれば再発=手順③で再提案 / 無ければ手順④で「反映済み」確定 / 初回は N/A。**日付不明 note・判断に迷うセルは必ず出典 note 本文を確認**してから下す（カテゴリ粒度の誤読防止）。

### 1.6 `learnings.md` の棚卸し（正式昇格・剪定＝二段昇格の後段）

[`docs/ai-dlc/learnings.md`](../../../docs/ai-dlc/learnings.md)（Gate 3 の `retro-surface` が機械追記した未昇格の経験則）を読み、各エントリを次に振り分けて手順③の提案に含める（`learnings.md` が存在しない／空なら N/A）:

| 振り分け | 行き先 | 後処理 |
|---------|--------|--------|
| 繰り返し効く判断・方針 | 該当 `.claude/rules/*.md` へ正式昇格（人間承認後に AI が反映） | 昇格した行を `learnings.md` から除去 |
| grep/lint/型で機械判定できる不変条件 | sensor 化（`.claude/aidlc/src/sensors/`）の改善タスク案 | sensor 追加 PR 後に該当行を除去 |
| 効かない／陳腐化 | 剪定（削除） | 該当行を `learnings.md` から除去 |

- 正式昇格・剪定は**人間承認（手順③）が先・書き込みは後**（本スキルの重要原則どおり）。
- 除去結果は手順⑤の実施記録に「`learnings.md`: N 件昇格 / M 件剪定」の形で 1 行残す。

### 2. ハーネス剪定レビュー

- README の「ハーネス剪定レビュー」の 2 チェック（役立たなかったルール/Stage の簡素化・削除、
  繰り返し違反されたルールの hook / lint への機械化）を実施し、提案があれば手順③の提案に含める。

### 3. 改善タスク案の提示と承認【人間ゲート】

- 集約した Try と剪定レビュー結果を、**ハーネス改善タスク案（1 タスク = 1 PR）** に整理して提示する。
  各案に「対象 Try / 還流先ファイル / 変更概要」を添える。
- 人間の承認（採用 / 不採用 / 保留）を得る。**承認を得るまで手順④⑤に進まない**。

### 4. 各 note のステータス欄更新

承認結果に従い、各 retro note の「フロー改善アクション」表のステータス欄を更新する:

| 承認結果 | ステータス |
|---------|-----------|
| 採用（Issue 起票） | Issue化 |
| 採用（改善 PR 済み） | 反映済み(PR#) |
| 不採用 | 見送り(理由) |
| 保留 | 未対応のまま（次回棚卸しで再集約） |

### 5. 棚卸し実施記録への追記

- README の「棚卸し実施記録」に 1 行追記する（日付・対象 note 件数・主な採用 Try）。
  これが次回の「1 か月経過」の計算起点になる。
- 記録には**前回 Try の再発チェック結果**（効いた / 未達 → 再提案）も 1 行で含める。

## 報告フォーマット

- 集約した未対応 Try の一覧（出典 note つき）
- 前回 Try の効果測定結果（効いた / 未達で再提案した Try の一覧）
- 剪定レビューの結果（提案あり / なし）
- 改善タスク案ごとの承認結果と、更新した note・記録の一覧
