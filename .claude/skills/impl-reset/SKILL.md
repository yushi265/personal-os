# 実装一括リセット手順

「実装」と「ハーネス」を分離して、実装だけを安全に取り除く。
**削除は不可逆になり得るため、対象リストの人間承認なしに実行しない。**

## 0. 前提整理

- 残すもの（ハーネス・基盤）: ルート/各レイヤーの `CLAUDE.md`・`docs/`（`docs/spec/` の扱いは手順 2 で判断）・
  `.claude/`（rules / agents / skills / settings）・lefthook 等のツールチェーン設定・各レイヤーの初期スキャフォールド
- 消すもの（実装）: 追加したアプリコード・テスト・マイグレーション/seed・生成物・実行環境（例: compose）のサービス追加分

## 1. 削除対象の洗い出し

実装がどのコミット/パスに入っているかで方法を選ぶ。

```bash
# A) 実装がブランチ/コミットで分離されている場合（推奨状態）
git log --oneline <base>..<impl-branch>          # 実装コミットの確認
git diff --stat <base>..<impl-branch>            # 対象ファイル一覧

# B) 分離されていない場合（パスで特定）
git log --oneline -- <プロジェクトの実装ディレクトリ...>   # 例: src/ app/ 各レイヤーの実装パス
```

- 判断に迷うファイル（共有設定・ハーネス両用ファイル）は「残す」に倒し、個別に列挙して確認する。

## 2. 【人間ゲート】対象リストの承認

- 「削除するファイル/コミット」「残すファイル」「`docs/spec/` の実装 spec を残すか（学びの記録として残す/実装と道連れにする）」を提示し、**承認を得てから**次へ。

## 3. リセット実行

```bash
# まず実行環境を畳む（データボリュームも破棄）
<実行環境の停止コマンド>   # 例: docker compose down -v --remove-orphans

# A) ブランチ分離されている場合: ベースへ戻るだけ（実装ブランチは残置 or 削除を選択）
git switch <base-branch>

# B) パス指定で取り除く場合: 新しいリセットブランチ上で削除コミットを作る
git switch -c <reset-branch>
git rm -r <承認済みの削除対象パス...>
# 実装で書き換えた共有ファイル（例: 実行環境の compose 定義）はベース時点の内容へ戻す
git checkout <base-commit> -- <実装で書き換えた共有ファイル...>
git commit  # コミット前にファイルリストを再提示して承認を得る
```

- `git push -f` や `git reset --hard` での履歴改変はしない（`-f` 系コマンド禁止。削除も「削除コミット」として歴史に残す）。

## 4. リセット後の検証

```bash
git status --short                  # 意図しない残骸・消し漏れがないか
ls <プロジェクトの実装ディレクトリ...> 2>/dev/null    # 実装が消えているか
<実行環境の設定検証コマンド>          # 実行環境がベース状態として妥当か（例: docker compose config --quiet）
<プロジェクトのビルド・テストコマンド>  # ベース状態でビルド/テストが通るか
```

- ハーネス（`.claude/` / docs / CLAUDE.md）が無傷で残っていることを確認する。
- node_modules / 生成キャッシュ等のローカル残骸が気になる場合のみ `git clean -nd` で**dry-run 確認**し、
  対象を人間に提示してから `git clean -d -i`（対話モード）で消す。`-f` での一括強制削除はしない。

## 禁止

- 人間承認なしの削除実行・`git clean -f`・`git reset --hard`・force push。
- ハーネス（CLAUDE.md / docs / .claude/）を巻き込んで消すこと。
- 「ついで」の整理（リセットと無関係なリファクタを混ぜない）。
