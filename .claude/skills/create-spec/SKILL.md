# 実装 spec 作成手順

`docs/spec/<TICKET>-<kebab-case-feature>/` ディレクトリを `docs/spec/_TEMPLATE/` 準拠で作成する。
spec は実装の正本（AC・対象範囲・制約）。以降の実装は spec を契約として進め、乖離したら spec を先に更新する。

## spec の構成（チケット単位ディレクトリ + レイヤー別ファイル）

```
docs/spec/<TICKET>-<kebab-case-feature>/
├── index.md     # 入口: 概要・対象範囲・ユニット計画・AC・アーキテクチャ・横断エラー表・テスト戦略（テストケース一覧含む）・判断根拠（Why）
├── progress.md  # 揮発物: Stage 宣言・ゲート承認状態・現在位置・実装タスク計画（完了時に除去）
└── <layer>.md   # 関与レイヤーごとに 1 ファイル（レイヤー名・分割はプロジェクトが定義）。例:
                 #   data.md    = スキーマ/マイグレーション増減・データ境界（アクセス制御）設計・seed
                 #   service.md = API パス増減・対象ドメイン・入出力・エラー・実装配置
                 #   ui.md      = 画面パス増減・使用する API・コンポーネント設計・ルーティング
```

- 設計判断（Why・採用案/却下案・トレードオフ）は `index.md` の「判断根拠 / 未決事項」が正本、UI/UX 方針は表示層 spec（例: `ui.md`）の「UI/UX 方針」が正本。
  方針と契約（具体値: スキーマ / 入出力 / パス）を同一ファイル内で書き分ける。
- **AC は `index.md` が正本**。各レイヤー .md（例: `data.md` / `service.md` / `ui.md`）は「担保 AC 番号 + 引用文」を持つ（[spec-conformance-reviewer](../../agents/spec-conformance-reviewer.md) が引用整合を機械チェック）。
- **AI 実装エージェントは自レイヤー .md 単独で実装可能なように書く**（テストケース詳細・異常系挙動はレイヤー .md に置く・index.md はサマリのみ）。

- **関与しないレイヤーのファイルは作らない**（例: 表示層のみのタスクは index.md + `ui.md`）。
- `progress.md` は契約ではなく**揺れる作業ファイル**（AI-DLC の中断 → 再開用）。`index.md` の AC を重複させず、機能完了時（Gate 3 / merge 前）に除去する（[ai-dlc-flow](../ai-dlc-flow/SKILL.md) の「progress.md による中断 → 再開」を参照）。
- `index.md` が常に入口。レイヤー間で共有する契約値（seed の固定 UUID 等）は所有レイヤーのファイルを正本にし、他レイヤーからは参照する。

## 手順

1. **テンプレートを読む**: `docs/spec/_TEMPLATE/`（index / progress / 各レイヤー .md）。
2. **横断ルールを読む**: `.claude/rules/`（spec-driven / simplicity / testing）。
3. **対象レイヤーの規約を読む**: 対象になる各レイヤーの規約 docs（プロジェクトで用意）。
   spec はハーネスの設計方針に**従う**。spec で方針を上書きしない。矛盾を見つけたら人間へ報告する。
3.5. **既存実装を調査する（現実装リバースエンジニアリング）**: **まず [`docs/ai-dlc/codekb/`](../../../docs/ai-dlc/codekb/) の該当ファイルを読み、調査範囲を「codekb に無い部分 + 鮮度が怪しい部分」の差分に絞る**（記述は仮説・`参照:` パスの実在確認必須）。その上で、関連する既存ドメイン・既存スキーマ・既存 API 契約・共有コンポーネント／ユーティリティを調査し、**再利用できるもの / 差分 / 衝突 / 依存**を洗い出す。[self-review](../self-review/SKILL.md) の観点C（再発明禁止・既存破壊禁止・命名/配置の一貫）を**着手前**に先取りする狙い。対象は**このプロジェクト内の現実装**（外部・旧システムの解析ではない）。結果は `index.md` の「既存実装との関係」「判断根拠」に記録する（既存資産が薄ければ簡潔に＝YAGNI）。**委譲メカニクス**（Stage 2a・`Explore` 並列起動・統合）は [ai-dlc-flow](../ai-dlc-flow/SKILL.md) の Stage 2 を参照。
4. **レイヤー間インターフェースを先に確定させる**（複数レイヤー実装の要）。関与する各レイヤーの .md に具体値で書く（レイヤー名・分類はプロジェクトに合わせる。以下は例）:
   - データ層（例: `data.md`）: スキーマの増減表・マイグレーション定義（構造・制約・インデックス）・データ境界（アクセス制御）設計・seed 定義
   - サービス層（例: `service.md`）: API パス増減表・対象ドメイン・リクエスト/レスポンスの具体値・エラー形式・ステータス・権限チェック仕様
   - 表示層（例: `ui.md`）: 画面パス増減・使用する API 一覧・コンポーネント設計（サーバー/クライアント境界）・ルーティング仕様・セッション/トークンの受け渡し形式。
     **表示層が関与する時は「UI/UX 方針」（画面フロー・主要操作・状態設計〔初期/ローディング/空/エラー/成功〕・レスポンシブ/アクセシビリティ・既存デザインシステムとの整合）も書く**（Stage 1 で確認した内容を反映。UI の不確実性が高い場合は画面モックを作って合意してから契約を固める）。
   - 加えて各レイヤー .md に「担保 AC（引用文・index.md とリテラル一致）」「テストケース（技法注記付き）」「異常系挙動」3 節を必ず埋める
   - 表示層が関与する時は「レスポンシブ / アクセシビリティ」節も必須（空のままにしない）
5. **AC を観測可能な形で書く**（`index.md`）: 「テストで判定できる」粒度。正常系・異常系（認証・権限・not found・外部障害 等）・境界値を含める。
6. **テスト戦略をマトリクスで書く**（`index.md`）: `.claude/rules/testing.md` の「テスト種別の分担」基準
   （正常系フロー・異常系・境界値・データ境界分離 = レイヤー内結合 / ロジック・変換 = 単体）に従い、
   AC × 種別 × レイヤーの担保場所を表で確定する。
6.5. **テストケース一覧を導出する**（**各レイヤー .md** の「テストケース」節）: 技法マッピング表（[testing.md](../../rules/testing.md)）で
   AC ごとにケースを導出し、**各項目の先頭に適用技法を注記**して列挙する（記法は [tdd-cycle](../tdd-cycle/SKILL.md)）。`index.md` にはケース詳細を書かず**テストマトリクス（AC × 種別 × レイヤー）サマリのみ**残す。
   これは **Gate 2 で人間に提示**してテスト設計をレビュー可能にするための一覧（Gate 2 委任時も同様に提示する。変わるのは承認待ちの有無のみ: [risk-tiers.md](../../rules/risk-tiers.md)）。Stage 3（RED）はこの一覧を起点に進め、発見したケースは progress.md に追記する。
7. **エラー・ログ方針を明記する**（`index.md` に横断表）: 各レイヤーのエラー設計方針に従い、異常系の挙動（エラーコード・ユーザー向け表示・ログ/監査）を表で確定する。
8. **やらないこと（スコープ外）を明記する**（`index.md`）: YAGNI。将来要件の先取り禁止。
9. **設計判断（Why）と未決事項を書く**（`index.md` の「判断根拠 / 未決事項」）: 採用したアプローチ・却下した代替案とその理由・受け入れるトレードオフ／リスクをまとめる（trivial な判断は理由だけで足りる＝YAGNI）。判断できない未決点は列挙し、**人間に確認してから**実装に進む。未決点が 4 個以上 / 非同期回答になりそうな場合は `questions.md`（雛形: `docs/spec/_TEMPLATE/questions.md`）に列挙する（使い分け表は [ai-dlc-flow](../ai-dlc-flow/SKILL.md) Stage 0+1。空 `[Answer]:` が残ったまま Gate 2 に進まない）。
10. **（非trivial時）実装タスクを順序分解する**: 確定した契約・AC をもとに、ユニットを順序付きの実装タスク（ボルト）へ分解し、各タスクへ「レイヤー・カバーAC・依存」を割り付けて `progress.md` の「実装タスク計画」に**台帳記法**（`- [ ] T<n> [レイヤー] AC-… 依存:… — タイトル`。雛形 `_TEMPLATE/progress.md` 参照）で書く。この記法なら `pnpm -C .claude/aidlc ledger` が未消化・未割当 AC・依存違反を機械検査できる。これは**実装の段取り**（Stage 3/4 の実装順）であり、**PR の単位は固定しない**（Gate 3 で人間が決める。[task-and-pr.md](../../rules/task-and-pr.md)）。trivial は不要。

## チェックリスト

- [ ] spec ディレクトリ単体で実装判断が完結する（外部ドキュメントを辿らず実装できる）
- [ ] 関与レイヤーのファイルが揃い、関与しないレイヤーのファイルが無い
- [ ] `index.md` の「判断根拠 / 未決事項」にアプローチ（採用案/却下案・トレードオフ）が書かれている
- [ ] 表示層が関与する時は表示層 spec（例: `ui.md`）に UI/UX 方針（状態設計含む）が書かれている
- [ ] レイヤー間インターフェースが具体値（スキーマ / 入出力 / パス等）で書かれている
- [ ] AC が観測可能（テスト化できる）で、テストマトリクスに担保場所が割り付けられている
- [ ] テストケース一覧が技法注記付きで導出され、Gate 2 で提示できる
- [ ] チケットが複数ユニットに割れる場合、ユニット計画が `index.md` に記録されている
- [ ] エラー・ログの挙動が異常系含めて確定している
- [ ] スコープ外が明記されている
- [ ] ハーネス（`.claude/rules/` / 各 `*/docs/`）と矛盾していない
- [ ] AC が `AC-1` `AC-2` のように番号付けされている
- [ ] 各レイヤー .md に「担保 AC（引用文）」「テストケース」「異常系挙動」3 節が埋まっている
- [ ] レイヤー .md の AC 引用文が index.md の AC 文言とリテラル一致している
- [ ] 表示層が関与する時は「レスポンシブ / アクセシビリティ」節が埋まっている（対象端末・主対象ブレークポイント・タブレット方針・スマホ方針・a11y 最低限）
