# ハーネス設計判断（なぜこの構成か）

本書はこの AI-DLC ハーネス（`.claude/` 配下 + 各 `CLAUDE.md` + 各 `docs/`）の構成上の決定と理由・検討した代替案を残す履歴的資料です（ADR は設けず本書に集約）。

> AI が日常作業で参照する必要はない（ハーネスの使い方は [`.claude/README.md`](../.claude/README.md)、横断ルールは [`.claude/rules/`](../.claude/rules/) が正本）。本書はハーネスを後から見直す人間・新規参入者向け。

## 決定事項

| 項目 | 決定 |
|------|------|
| 守るガードレール | ① spec-first ② KISS/YAGNI（シンプルさ最優先） ③ TDD 80%+（＋ツール統一はベースライン）。設計規約（レイヤー境界・セキュリティ等）は同梱せずプロジェクトが必要に応じて追加 |
| spec 参照戦略 | 実装リポ内 `docs/spec/<TICKET>-*/` に薄い spec を置き、実装に必要な制約を要約して自己完結させる |
| AI-DLC フロー形態 | **単一適応フロー**（2 トラック廃止）。冒頭の**Stage 宣言**で Stage の実行/スキップを決め、🔒必須 Stage（TDD・静的解析・セルフレビュー・コミットゲート）は破れない。spec（Stage 2）のみ🔓条件付き。spec 省略基準は `spec-driven.md` に明文化。中断 → 再開は spec ディレクトリの `progress.md`（揮発物・完了時除去）。adaptive workflow / state の考え方を KISS/YAGNI に合わせ軽量移植（audit ログ・独立計画 Stage・Mermaid 成果物化は**不採用**） |
| 採用パターン | 薄い `CLAUDE.md`（アダプタ）＋ `docs/` ＋ `.claude/rules/`（横断）。サブツリー分割する場合の固有規約は各サブツリーの `docs/` 直下 |
| 機械強制と AI 誘導の二層 | 破ってはいけない不変条件は**機械強制**（lefthook / gitleaks / カバレッジ / コミットゲート）に、判断を要する規約は**AI 誘導**（`.claude/rules/` / skills / self-review）に置く。AI が無視できる助言と、物理的に止まるゲートを役割分担する |
| engine（aidlc CLI）は advisory | 進行管理・台帳検査・sensor 等の補助ツールは advisory（助言）とし hard stop にしない。機械強制は最小の不変条件（コミットゲート等）に絞り、フローの柔軟性を殺さない（KISS/YAGNI） |

## 検討した代替案

| 案 | 内容 | 不採用理由 |
|----|------|-----------|
| 一枚岩 `CLAUDE.md` | 全ルールを 1 ファイルに集約 | 肥大化・コンテキスト圧迫・更新衝突 |
| 重厚マルチエージェント | 層ごと専用 agent 必須・subagent 駆動前提 | 現段階ではオーバー（YAGNI 違反） |

## やらないこと

- サブツリー固有ルールのルート上書き（各サブツリーの `CLAUDE.md` に委譲）。
- ルールを 1 ファイルに詰め込みすぎること。薄い `CLAUDE.md` ＋ `docs/` ＋ `.claude/rules/` に役割分担する（重複転記しない）。
- **未確定**の領域を、確定したものとして断定記述すること（先取りしない）。
- 必要になっていない仕組みの先行実装（YAGNI）。
