# codekb — 既存実装調査の永続コード知識ベース

毎ボルトの Stage 2a（既存実装調査）が「フルリポ再調査」になるのを防ぐための**差分化ヒント集**。
Explore サブエージェントへの委譲時に該当ファイルを添付し、調査範囲を「codekb に無い部分 + 鮮度が怪しい部分」に絞る。

> **codekb は各プロジェクトが自分の実装スナップショットを育てる場所**。この README が持つのは
> ファイル構造・鮮度規約・分割単位・剪定基準の**枠**だけで、実データ（`shared.md` 等）はプロジェクトが追記する。

## 鮮度規約（読む側・書く側の契約）

1. **codekb はヒント・仮説**（正本はコードと spec）。読む側（Explore）は記述を鵜呑みにせず、
   **`参照:` パスの実在を必ず確認**する。実在しない・内容が違う場合は codekb の側が stale。
2. `最終更新` の commit SHA と HEAD の乖離が大きい節は **stale 前提**で差分調査する。
3. 更新は **Gate 3 の 1 ステップ**（retro-surface と同列）: そのボルトで判明した
   新 API・新テーブル・新部品・新しい罠を該当ファイルへ追記し、`最終更新` を書き換え、
   **実装と同一コミット**に含める（spec 更新と同じ扱い）。

## ファイル構造（固定・機械チェック可能に）

```markdown
# codekb: <対象>
## 公開インターフェース   … レイヤーが公開する IF（API パス / スキーマ型 / ルーティング等）（参照: <path> 必須）
## 主要データ構造         … テーブル・スキーマ一覧と制約（アクセス制御含む）（参照: <path>）
## 再利用可能な部品       … ユーティリティ・共有コンポーネント（参照: <path>）
## 既知の罠               … retro / learnings 由来の 1 行 + 出典リンク
## 最終更新               … <TICKET> / commit <SHA> / YYYY-MM-DD
```

## ファイル分割の単位

- 初版は `shared.md`（横断）のみで開始し、**実需が出た単位で** `<layer>.md` / `<layer>-<domain>.md`
  （例: `service-<domain>.md` / `ui.md`）を追加する（Rule of Three。空の器だけ先に作らない）。

## 剪定基準

- stale 記述による誤誘導（手戻り）が調査削減益を上回ったら、剪定 or 自動生成へ方針転換する
  （retro-triage で判断。運用実測は retro note に記録）。
